The Grammar

Why expressions need a stronger formalism than the Scanner's, and what Lox's actual expression grammar looks like.

Regular grammars (the Scanner's level) can recognize patterns but can't count — they have no way to track arbitrary nesting depth. Expressions nest without bound (((((1))))), so they need a context-free grammar: a set of named rules (productions) made of terminals (literal tokens) and nonterminals (references to other rules). A rule referencing itself is what makes infinite nesting possible from a finite set of rules.

expression → literal | unary | binary | grouping ;
literal    → NUMBER | STRING | "true" | "false" | "nil" ;
grouping   → "(" expression ")" ;
unary      → ( "-" | "!" ) expression ;
binary     → expression operator expression ;
operator  → "==" | "!=" | "<" | "<=" | ">" | ">="
            | "+"  | "-"  | "*" | "/" ;
Known issue, deliberately left in: binary's rule encodes no precedence or associativity. 6 / 3 - 1 is genuinely ambiguous as written — it could mean (6/3)-1 or 6/(3-1). Fixing that is Chapter 6's job, by stratifying this one rule into a cascade of precedence levels.
03

The Expr Class Hierarchy

Turning the grammar into Java — and why these classes hold no behavior at all.

Why one class per production, not one generic node

A single Expression class with a generic list of children would work, but throws away type safety. Instead: one abstract base (Expr), one subclass per production (Binary, Grouping, Literal, Unary), each with only the fields it needs. The compiler itself then catches mistakes — there's no second operand to mistakenly read off a Unary, because the field doesn't exist.

"Disoriented objects" — why no methods live here

Expr is built by the Parser and consumed by the Interpreter (and later, a Resolver). It doesn't belong to either domain, so it holds no behavior — pure data, existing only to let those phases communicate. This is a deliberately functional-style choice, borrowed into an otherwise object-oriented codebase.

abstract class Expr {
  static class Binary extends Expr {
    Binary(Expr left, Token operator, Expr right) {
      this.left = left;
      this.operator = operator;
      this.right = right;
    }
    final Expr left;
    final Token operator;
    final Expr right;
  }
  // Grouping, Literal, Unary follow the same shape...
}

final fields — and why it matters more here than for Token

One tree gets read by multiple independent consumers over its lifetime — the Interpreter today, a Resolver later. If any reader could mutate a node mid-walk, every other consumer's view of the program's structure could shift underneath it. final makes that bug class structurally impossible. Same principle as Token: tokens are facts about the past; Expr nodes are facts about the program's structure — neither should change once built.

abstract, precisely

abstract class Expr blocks new Expr() directly — there's no such thing as a bare expression, only a specific kind. Once accept() is added (next section), abstract does its second job too: forcing every subclass to implement it, or fail to compile.

04

GenerateAst.java

A build-time tool, not part of jlox itself — it writes Expr.java for us.

Each tree type collapses into one string: "Binary : Expr left, Token operator, Expr right". The script splits on : for the class name vs. fields, then , for individual fields, and mechanically emits a full nested class — constructor, field assignments, field declarations — for each one. baseName is a parameter, not hardcoded, because this exact script gets reused later to generate Stmt.java too.

Fix the generator, not the output. Every bug found in Expr.java (capitalized field names, a missing semicolon) was fixed in GenerateAst.java and regenerated — editing the generated file directly would just get silently overwritten on the next run.

The Visitor Pattern

How to attach behavior to classes that deliberately hold none.

The Expression Problem

Every operation-on-every-type is a cell in a table — rows are types, columns are operations:

print()interpret()resolve() (later)
Binarycodecodecode
Groupingcodecodecode
Literalcodecodecode
Unarycodecodecode

OOP's grain bundles by row (a class, with methods) — easy to add types, hard to add operations. Functional pattern-matching bundles by column — the opposite tradeoff. Lox's trees need more operations over time far more often than more types — so the Visitor pattern is used specifically to fake the functional, column-oriented grain inside Java.

The mechanism — accept() + Visitor<R>

interface Visitor<R> {
  R visitBinaryExpr(Binary expr);
  R visitGroupingExpr(Grouping expr);
  R visitLiteralExpr(Literal expr);
  R visitUnaryExpr(Unary expr);
}

// inside Expr.Binary:
<R> R accept(Visitor<R> visitor) {
  return visitor.visitBinaryExpr(this);
}

expr.accept(visitor) resolves in two genuinely different stages:

Stage 1 — real, runtimeStage 2 — fixed, compile-time
The JVM follows expr's actual class (via a hidden reference every object carries) to find which subclass's accept() runs.That subclass's accept() body calls one specific, hardcoded visit method name — baked in permanently when GenerateAst.java wrote the file.

Distinct method names (visitBinaryExpr, not an overloaded visit(Binary)) make it unmistakable that polymorphism — not overload resolution — does the routing.

Why <R> is generic

AstPrinter wants String back. The future Interpreter will want Object. Visitor<R> fixes R once, the moment a class implements it. accept<R> re-resolves R fresh at every call site, based on whichever visitor is actually passed in — letting one accept(), written once, correctly serve every visitor that will ever exist.

06

AstPrinter

The first real Visitor — proof the whole mechanism actually works.

class AstPrinter implements Expr.Visitor<String> {
  String print(Expr expr) { return expr.accept(this); }

  public String visitBinaryExpr(Expr.Binary expr) {
    return parenthesize(expr.operator.lexeme, expr.left, expr.right);
  }
  public String visitGroupingExpr(Expr.Grouping expr) {
    return parenthesize("group", expr.expression);
  }
  public String visitLiteralExpr(Expr.Literal expr) {
    if (expr.value == null) return "nil";
    return expr.value.toString();
  }
  public String visitUnaryExpr(Expr.Unary expr) {
    return parenthesize(expr.operator.lexeme, expr.right);
  }

  private String parenthesize(String name, Expr... exprs) { ... }
}

Test input — built by hand, since the Parser doesn't exist yet:

Expr expression = new Expr.Binary(
    new Expr.Unary(minusToken, new Expr.Literal(123)),
    starToken,
    new Expr.Grouping(new Expr.Literal(45.67)));

System.out.println(new AstPrinter().print(expression));
// → (* (- 123) (group 45.67))
deep dive

The Questions That Actually Mattered

Once the code compiled and ran, the real question was: what is actually happening when this runs?

Q·1 Is scanToken() / is this tree-walk recursive?

Scanning never was — scanTokens()'s while-loop is plain iteration, and the comment-skipping loop only calls advance()/peek(), never itself. Lexemes don't nest, so nothing in the Scanner needed real recursion.

This tree, by contrast, is genuinely recursive — just indirectly (mutual recursion): parenthesize() never calls itself by name, but it calls accept(), which calls a visit*Expr(), which — for anything except a Literal — calls parenthesize() again. The cycle closes through three hops instead of one.

Q·2 What does "the visitor" actually refer to?

One concrete object — a single AstPrinter instance, created exactly once (new AstPrinter()) — that gets handed, unchanged, to every accept() call across the entire tree. It's a roving piece: the tree's nodes are fixed and separate (final fields, built once); the visitor is the one thing that travels through all of them, deciding what to do at each stop.

Q·3 How does expr.operator.lexeme resolve?

Plain left-to-right chained field access, nothing more. expr.operator reads the Token field off the Binary node; .lexeme then reads the String field off that Token. It compiles without a getter only because lexeme was declared package-private back in Token.java, and AstPrinter lives in the same package.

Q·4 Where does left vs. right actually get separated in code?

Two different moments, not one. Structurally, the separation is permanent the instant Binary's constructor runs — two distinct, named, final fields. Procedurally, the order they get visited in is just whatever order they're written in visitBinaryExpr's one line: parenthesize(op, expr.left, expr.right). Varargs pack them into an array in exactly that order; the loop walks the array start to end. Swap that line's argument order, and the tree doesn't change — only the reading order does.

Q·5 What does the parenthesized output actually mean?

It's not Lox syntax and nothing has been evaluated — it's a structural X-ray. The position of each paren directly reveals the tree's real shape: the symbol right after ( says what kind of node it is, and everything up to the matching ) is exactly that node's operand(s). No precedence has to be "remembered" while reading it — the nesting is the precedence, made visible. This becomes the actual debugging tool once the Parser exists: feed it real Lox, and misplaced parens in the output reveal precedence bugs instantly.

Q·6 Two different "this" — how does visitor.visitBinaryExpr(this) actually resolve?

There are genuinely two separate this values in play, in two different files. Inside AstPrinter's own methods, this means the AstPrinter. Inside Expr.Binary's accept(), this means the Binary node — a completely different object — and the visitor is received there under a different name entirely: visitor. The line reads: "call visitBinaryExpr on the visitor I was handed, passing myself in."

Picking which accept() runs is real, runtime polymorphism (the JVM follows a hidden class-reference every object carries). Picking which visit method that accept() calls is not dynamic at all — it's a fixed string, baked into the file once by GenerateAst.java.

Q·7 Is this DFS? And what kind, precisely?

Yes, by necessity — a call can't return before everything it called has returned, so there's no way to get breadth-first behavior out of plain recursion like this. More precisely: dispatch is pre-order (a parent's visit*Expr is entered before its children's), but value-completion is post-order (a parent can't finish building its string until both children have already finished theirs). Binary is visited first, but it's the last one to actually finish.

Q·8 What is expression, really — an analogy?

It's a reference to exactly one object — the root node — nothing more. The "tree" isn't a bundled container; it's a web of separate objects holding references to each other, reachable only by walking outward from that one root. The closest real analogy is a linked list, generalized: instead of one anonymous next per node, each Expr subclass has its own specific, named outgoing references (left/right, or just right, or none at all for a Literal).

One Call, Start to Finish

new AstPrinter().print(expression) — every frame, tracked by depth, until it terminates.

0main(): new AstPrinter().print(expression)
1print(Binary-Root)
2Binary-Root.accept(P)
3P.visitBinaryExpr(Binary-Root)
4parenthesize("*", Unary-X, Grouping-Y)
5Unary-X.accept(P)
6P.visitUnaryExpr(Unary-X)
7parenthesize("-", Literal-A)
8Literal-A.accept(P)
9P.visitLiteralExpr(Literal-A) → "123" (base case)
·↑ unwinds: "123" → "(- 123)" → returns to outer parenthesize, builder = "(* (- 123)"
5Grouping-Y.accept(P) — loop item 2
6P.visitGroupingExpr(Grouping-Y)
7parenthesize("group", Literal-B)
8Literal-B.accept(P)
9P.visitLiteralExpr(Literal-B) → "45.67" (base case)
0↑ full unwind → "(* (- 123) (group 45.67))" → println()
print / entry accept (dispatch) visitBinaryExpr visitUnaryExpr visitGroupingExpr base case (Literal)
Max depth: 9. Exactly 2 × levels − 1, since every level costs two frames (accept() then visit*Expr()) before recursing further or hitting a Literal. Termination is guaranteed because Expr describes a tree — no cycles are possible, since every field was built before the node holding it, and a cycle would require pointing back to something that doesn't exist yet.

Notebook Corpus, Condensed

The six handwritten blocks, for quick reference alongside the notebook itself.

Block 1 — CFG Notation
terminal = literal token, an "end point". nonterminal = named reference to another rule. Self-reference in a rule = how nesting becomes possible from finite rules. Notation: → defines, "quoted"=terminal, lowercase=nonterminal, | alternation, (...) grouping, * zero-or-more, + one-or-more, ? optional.
Block 2 — Lox's Expression Grammar
expression → literal | unary | binary | grouping ; grouping/unary are self-referential -> this is what allows nesting. binary's rule is ambiguous on precedence as written (fixed in Ch.6).
Block 3 — The Expr Hierarchy
One abstract Expr + one subclass per production (Binary/Grouping/Literal/Unary). "Disoriented objects": no methods, pure data -> Expr belongs to neither the Parser nor the Interpreter, so it can't naturally hold either's behavior. final fields: a tree, once built, must not change under its later readers.
Block 4 — GenerateAst.java
Build-time tool, separate package (.tool, not .lox). Mini-DSL string per type: "ClassName : type name, type name". baseName is a parameter so the same script can later generate Stmt.java too. Fix the GENERATOR, never the output.
Block 5 — The Visitor Pattern
The Expression Problem: types(rows) x operations(columns). OOP eases adding rows; functional eases adding columns. Visitor fakes the functional grain inside Java: Visitor<R> (one method per type) + Expr.accept(visitor) on each subclass, calling its own named visit method. Dispatch = polymorphism on accept() (real, runtime). WHICH visit method gets called = fixed, compile-time, baked in by the generator.
Block 6 — AstPrinter
First real Visitor<String>. print() = expr.accept(this) — "this" IS the visitor. parenthesize() is the recursive engine (mutual recursion via accept/visit*Expr), bottoming out at Literal (no subexpressions = base case). Output is a Lisp-style, fully-parenthesized X-ray of the tree's real shape — not Lox syntax, not a computed value. Used to visually verify the Parser's precedence handling once Chapter 6 exists.
