Ch.7 · B1

Runtime Values — the type bridge

Lox is dynamically typed; Java is statically typed. The bridge is java.lang.Object.

Lox typeJava representationWhy
any valueObjectCommon supertype of everything in Java
nilnullJava's null reference
true/falseBooleanBoxed primitive
numberDoubleALL Lox numbers — no int/float split
stringStringDirect Java String

The literal → value chain (one conversion, three phases)

SOURCE TEXT:  "123.45"
      │
      ▼  SCANNER — Double.parseDouble("123.45") → Double(123.45)
      │            ← THE ONE AND ONLY CONVERSION MOMENT
      ▼
Token.literal = Double(123.45)       [Object field, final]
      │
      ▼  PARSER — previous().literal → reads Double off the token
      │
Expr.Literal.value = Double(123.45)  [Object field, final]
      │
      ▼  INTERPRETER — return expr.value
      │
Double(123.45)  ← same object, zero re-conversion
Key insight: The same Java object travels the entire pipeline. Java passes object references, not copies — Token.literal, Expr.Literal.value, and the Interpreter's return value are all pointing at the same Double in memory.

Literal vs Value — a real distinction

A literal is syntax that appears in source code (123, "hello", true). A value is a runtime entity produced by evaluation. 1 + 2 produces the value 3, but 3 is never a literal anywhere in that source. Expr.Literal exists for literals only.

evaluate() + Literal + Grouping

The recursive engine and the two simplest visit methods.

// The engine — threads `this` down every accept() call
private Object evaluate(Expr expr) {
  return expr.accept(this);

@Override

@Override
public Object visitGroupingExpr(Expr.Grouping expr) {

Ch.7 · B3

visitUnaryExpr + Truthiness

The first visit method that does real work — introduces dynamic type casting and Lox's truthiness rule.

@Override
public Object visitUnaryExpr(Expr.Unary expr) {
  Object right = evaluate(expr.right);   // children FIRST — post-order forced

  switch (expr.operator.type) {
    case BANG:  return !isTruthy(right);
    case MINUS: return -(double)right;   // Java cast — dynamic typing in one line
  }
  return null;   // unreachable
}

private boolean isTruthy(Object object) {
  if (object == null) return false;            // nil is falsy
  if (object instanceof Boolean) return (boolean)object;  // respect actual value
  return true;                                 // everything else is truthy
}

Lox's truthiness — Ruby's rule

ValueTruthiness in LoxContrast
nilFALSYsame as most languages
falseFALSYsame as most languages
0TRUTHYJS/Python make this falsy
""TRUTHYJS/Python make this falsy
any number/stringTRUTHY—
The cast (double)right is dynamic typing made visible. In a statically-typed language the compiler would verify right is a number before this code runs. In Lox, this check happens at runtime, every time this line executes. checkNumberOperand() wraps this to produce a clean Lox error instead of a raw Java ClassCastException.

Order in isTruthy() matters: the instanceof Boolean check must come before return true — otherwise Boolean(false) would incorrectly return true (it's non-null).

Ch.7 · B4–5

visitBinaryExpr + RuntimeError

The biggest visit method — four categories of operators, one special case, and the first Lox-specific error type.

Both operands always evaluated first — a semantic choice

public Object visitBinaryExpr(Expr.Binary expr) {
  Object left  = evaluate(expr.left);   // BOTH before the switch
  Object right = evaluate(expr.right);  // even if a type error follows
  switch (expr.operator.type) { ... }
}
This is a language semantics decision, not just implementation detail. say("left") - say("right") prints both "left" and "right" before erroring on the subtraction. We could check left's type first and skip evaluating right — but we chose not to.

Four operator categories

CategoryOperatorsInput → OutputType check
Arithmetic- / *Double → DoublecheckNumberOperands()
PLUS (special)+Double+Double OR String+StringOwn instanceof check
Comparison> >= < <=Double → BooleancheckNumberOperands()
Equality== !=any → Booleannone — any types allowed

Why + is special

case PLUS:
  if (left instanceof Double && right instanceof Double)
    return (double)left + (double)right;
  if (left instanceof String && right instanceof String)
    return (String)left + (String)right;
  throw new RuntimeError(expr.operator,
      "Operands must be two numbers or two strings.");

Every other binary operator blindly casts to double. + has two valid type combinations — it needs instanceof to decide at runtime which kind of + this is. Mixing types ("hi" + 1) is a RuntimeError — Lox deliberately rejects implicit coercion.

isEqual() — why not Java's ==

private boolean isEqual(Object a, Object b) {
  if (a == null && b == null) return true;   // nil == nil
  if (a == null) return false;              // nil != anything
  return a.equals(b);                         // value equality, not reference
}

Java's == compares references (new Double(1.0) == new Double(1.0) is false). .equals() compares values. Guard against null.equals() → NullPointerException with the explicit null checks first.

RuntimeError — hiding the JVM from Lox users

class RuntimeError extends RuntimeException {
  final Token token;   // which operator caused it — for line reporting
  RuntimeError(Token token, String message) {
    super(message);
    this.token = token;
  }
}

Raw Java ClassCastException would expose implementation details. RuntimeError wraps the offending token so we can report [line 3] Operand must be a number. — a Lox error, not a Java stack trace. Same exception-as-stack-unwinder principle as ParseError.

stringify() — trimming the .0

private String stringify(Object object) {
  if (object == null) return "nil";
  if (object instanceof Double) {
    String text = object.toString();
    if (text.endsWith(".0")) text = text.substring(0, text.length() - 2);
    return text;   // 123.0 → "123", 123.4 → "123.4"
  }
  return object.toString();
}

Statements — the second syntax tree

Expressions compute values. Statements produce effects. Two separate hierarchies, enforced by the Java type system.

The fundamental split

ExprStmt
Producesa valuean effect
Visitor typeVisitor<Object>Visitor<Void>
Visit returnsthe computed valuenull always
Interpreter methodevaluate(Expr)execute(Stmt)
Parser returnssingle Expr (Ch.6)List<Stmt> (Ch.8)
Why Void (capital V)? Java's generics can't use lowercase void as a type argument due to type erasure. Void is a boxed placeholder — every statement visit method satisfies it by returning null. Weird, but harmless.

The new grammar

program   → statement* EOF ;
statement → exprStmt | printStmt | varDecl | block ;
exprStmt  → expression ";" ;
printStmt → "print" expression ";" ;
varDecl   → "var" IDENTIFIER ( "=" expression )? ";" ;
block     → "{" declaration* "}" ;

Parser call chain — two levels of looping

// OUTER LOOP — flat, iterative, one statement per iteration
List<Stmt> parse() {
  List<Stmt> statements = new ArrayList<>();
  while (!isAtEnd()) {
    statements.add(declaration());   // synchronize() lives here
  }
  return statements;
}

// INNER RECURSION — fires once per statement, builds the Expr subtree
private Stmt declaration() {
  try {
    if (match(VAR)) return varDeclaration();
    return statement();
  } catch (ParseError error) {
    synchronize();
    return null;
  }
}

Outer loop = program-level sequencer (flat, iterative, processes one statement per iteration). Inner recursion = expression-level parser (recursive descent, builds the Expr subtree inside each statement). These operate at completely different grammar levels.

execute() — statement equivalent of evaluate()

private void execute(Stmt stmt) {
  stmt.accept(this);   // same dispatch as evaluate(), different type
}
@Override
public Void visitExpressionStmt(Stmt.Expression stmt) {
  evaluate(stmt.expression);   // evaluate for side effects, DISCARD value
  return null;
}

@Override
public Void visitPrintStmt(Stmt.Print stmt) {
  Object value = evaluate(stmt.expression);
  System.out.println(stringify(value));
  return null;
}
Ch.8 · B3–4

Environment — the variable table

A map from variable names to runtime values. The first persistent state in jlox.

class Environment {
  final Environment enclosing;                      // parent chain (Block 5)
  private final Map<String, Object> values = new HashMap<>();

  // Three operations:
  void define(String name, Object value) {
    values.put(name, value);   // no existing-key check → redefinition allowed
  }

  Object get(Token name) {
    if (values.containsKey(name.lexeme)) return values.get(name.lexeme);
    if (enclosing != null) return enclosing.get(name);  // walk chain
    throw new RuntimeError(name, "Undefined variable.");
  }

  void assign(Token name, Object value) {
    if (values.containsKey(name.lexeme)) { values.put(name.lexeme, value); return; }
    if (enclosing != null) { enclosing.assign(name, value); return; }
    throw new RuntimeError(name, "Undefined variable.");
  }
}
OperationRequires prior existence?Used for
define()NO — creates or overwritesvar x = 10; declarations
get()YES — error if missingreading a variable (x + 5)
assign()YES — error if missingx = 20; reassignment
Keys are bare Strings, not Tokens. Two tokens at different source locations with the same name (x on line 1, x on line 5) must hit the same map entry. Raw string keys guarantee this — tokens carry position information we don't need for lookup.
Undefined variable = runtime error (not parse-time). Mutual recursion requires referring to functions before they're declared. Static detection would make isOdd() calling isEven() a parse error, even though it's perfectly valid code. Tradeoff: accept runtime detection to allow mutual recursion.

Ch.8 · B5

Scope — the Environment parent chain

Each block creates a new Environment pointing at its enclosing one. Variables are found by walking the chain inward to outward.

executeBlock — the mechanism

void executeBlock(List<Stmt> statements, Environment environment) {
  Environment previous = this.environment;    // save current scope
  try {
    this.environment = environment;           // switch to inner scope
    for (Stmt statement : statements) {
      execute(statement);
    }
  } finally {
    this.environment = previous;             // ALWAYS restore — even on error
  }
}
Why finally is critical: Without it, a RuntimeError thrown mid-block would skip the restore line, permanently leaving the Interpreter stuck in the inner scope. finally guarantees restoration regardless of how the block exits — normal completion, exception, or anything else.

Chain lookup — how variables find their scope

get() / assign() both walk the chain:
  check this.values → not found?
    check enclosing.values → not found?
      check enclosing.enclosing.values → ...
        until enclosing == null → RuntimeError

Innermost scope checked first — this is how shadowing works. A variable declared in the inner scope is found before the chain walk reaches the outer scope. The outer variable is untouched, just hidden.

Environment state for var a = "global"; { var a = "block"; print a; } print a;

GLOBAL ENVIRONMENT
"a" → "global"
enclosing: null (chain ends)
←
INNER (block scope)
"a" → "block"
enclosing: → GLOBAL

While inside the block: this.environment points at INNER. get("a") finds "block" immediately. After the block: finally restores this.environment to GLOBAL. get("a") finds "global". The inner "a" never touched the outer "a" — two separate map entries, two separate objects.

The Questions That Actually Mattered

Q·1 How is the value in Expr.Literal linked to what the Scanner produced?

The Scanner converts the raw text ("123.45") to a real Java Double via Double.parseDouble() — the one and only conversion moment. That Double is stored in Token.literal. The Parser reads it back via previous().literal inside primary() and passes it into new Expr.Literal(...), where it sits in Expr.Literal.value. The Interpreter reads that field in visitLiteralExpr and returns it. Same object, three containers, zero re-conversions.

Q·2 How does previous() retrieve the right token in primary()?

match(NUMBER, STRING) internally calls advance(), which increments current and moves the cursor past the token. By the time you're inside the if block, current has already moved forward. previous() reads tokens.get(current - 1) — exactly the token that was just consumed. This is why previous() exists at all: match() returns only a boolean, not which token matched. previous() retrieves the consumed token after the fact.

Q·3 How did we end up in visitBinaryExpr for Binary-Multiply when we called evaluate on Binary-Plus?

No routing logic anywhere — pure object references. Binary-Plus.left IS Binary-Multiply, a direct Java reference set at construction time by the Parser. When visitBinaryExpr(Binary-Plus) calls evaluate(expr.left), it's literally evaluate(Binary-Multiply). The JVM follows Binary-Multiply's hidden class reference, finds Expr.Binary's accept() override, which calls visitBinaryExpr(Binary-Multiply). No decision made — just following a reference that was set months ago (in Parser time).

Q·4 What is the mental map now that the Parser returns List<Stmt>?

Not "one tree" and not "a list of tokens" — a list of small trees, one per statement. Each Stmt node is the root of its own Expr subtree. The outer while loop in parse() sequences statements (flat, iterative). Each call to declaration() triggers the full recursive-descent expression parser to build one complete Expr subtree. The Interpreter processes each Stmt independently in order — they're separate units of execution, not connected by the tree itself.

Q·5 Why does the same operator token appearing twice in source code not collide in the environment?

Variable names are stored as bare strings, not tokens. values.put("a", ...) — the string "a" is the key. Two different token objects both holding lexeme = "a" hit the exact same map entry. The token's line number and source position are completely irrelevant to the lookup. This is an intentional design choice: logically, both uses of a refer to the same variable, regardless of where in the source they appear.

Q·6 In the scope trace, how does the print a inside the block find "block" and not "global"?

At the moment print a executes, this.environment points at INNER (switched there by executeBlock's assignment before the loop). visitVariableExpr calls environment.get(Token("a")) on whichever environment this.environment currently points at — which is INNER. INNER's values map contains "a" → "block", so it returns immediately without ever walking up to GLOBAL. The chain lookup only proceeds to the parent if containsKey returns false in the current environment.

Q·7 What is the difference between define() and assign()?

define() creates a new binding regardless of whether the name already exists — it's called for var x = 10; declarations. assign() requires the variable to already exist somewhere in the environment chain, and throws a RuntimeError if it doesn't — it's called for x = 20; reassignments. The distinction enforces Lox's requirement for explicit var declarations: you can't silently create a new global variable just by writing x = 10 without a preceding var x.

Q·8 Is the evaluation recursion a new kind compared to AstPrinter?

No — identical mutual recursion, different cargo. evaluate() → accept() → visit*Expr() → evaluate() is structurally identical to parenthesize() → accept() → visit*Expr() → parenthesize(). Same base case (visitLiteralExpr returns immediately with no further recursive call), same post-order completion (parent can't finish until children have returned). The only difference is what travels up the chain: AstPrinter carries String fragments; Interpreter carries Object runtime values.
